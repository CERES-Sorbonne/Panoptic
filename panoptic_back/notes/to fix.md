Now I have a complete picture. Here's everything that needs to be resolved

  before writing Project2:

  

  ---

  1. self.path vs self.db_path in ProjectDB

  SQLiteWriter.__init__ sets self.db_path, but ProjectDB.start() uses self.path

  to derive the folder name:

  Path(self.path).parent.name  # AttributeError — should be self.db_path

  Fix before anything else.

Fix: always use path.

  ---

  2. commits missing from IdRegistry

  ProjectDB.allocate_commits calls _handle_allocation('commits', ...) but

  IdRegistry has no commits field — it would raise ValueError at runtime.

  Looking at DataWriter, commit IDs are allocated via reader.get_max_commit_id()

   (i.e. MAX(id)+1), not through ProjectDB at all. Decision needed: either add

  commits to IdRegistry and route through ProjectDB, or remove allocate_commits

  and keep the MAX+1 approach in DataWriter. The MAX+1 approach is fine since

  there's only one write thread.

  Fix: remove the commit allocation it is a mistake. The commits are generated by the data_db to ensure absolute correctness

  ---

  3. check_same_thread on SQLiteWriter

  sqlite3.connect() without check_same_thread=False raises an error if the

  connection is used from a different thread than it was created on. If

  Project2.__init__ creates DataWriter on the main thread but the write executor

   uses it on the write thread, it crashes. Either add check_same_thread=False,

  or construct DataWriter inside write_executor.submit(...) during start().
	
Actually i want the writes to all happen inside one same thread loop. But maybe its ok on start to have it happens on some other place. Would it still be 100% safe? We also allow writing from another script so it seems like we can put check to False anyway ?
  

  ---

  4. DataReader threading strategy

  DataReader extends SQLiteReader which holds a single self.conn. Multiple

  FastAPI threads calling reads concurrently would all hit the same connection —

   unsafe. Decision needed: per-call context manager pattern, or thread-local

  storage?

  # Option: per-call (clean, safe, cheap with WAL page cache)

  def get_instances(self):

      with DataReader(self._data_db_path) as reader:

          return reader.get_instances()

  Fix: It should create on datareader per call that was the idea behind this strategy. Allow parallel multiple reads. Even parallel writes but because sqlite will queue the writes anyway no need to open even more connections.

  ---

  5. DataReader missing get_max_sequence()

  The DbWatcher needs it to seed last_seen_sequence on startup, but it's not

  implemented. Also get_commits_since(sequence) is missing — DataReader only has

   get_commits() with offset/limit, not sequence-based.

Fix: add it if needed

  ---

  6. Project folder layout — what files exist where?

  The note references Path(key.path) / 'project.db' for ProjectDB but DataWriter

   presumably opens a different file (the data DB). What are the exact

  filenames? Currently the old app uses a single panoptic.db. The new

  architecture has at least ProjectDB and DataWriter — are they one file or two?
  
  Fix: I imagine it like this. The projects lives in a folder. the folder has multiple dbs. project.db, data.db, media.db. The datawriter should only write to the data.db. The users are expected to use correct ids. Thoses are given by the project db. But the datadb doesnt need to know about project db.

  ---

  7. ProjectDB imports old models

  project_db.py imports Tag, Property, Instance from panoptic.models and File, 

  Folder, FileSource, Commit from panoptic.models.data. These are the old model

  classes. DataWriter and DataReader also use panoptic.models.data. Are these

  the same models or do they need reconciling?
  
  Fix: The new classes should use only new models if possible.

  ---

  8. Task manager for long operations

  The note says import_folder submits to a task manager, but no sync task

  manager exists yet. This doesn't block Project2 itself (the class can exist

  without import_folder), but the scope needs defining: is task management in

  Project2 or above it in Panoptic2?
  
  Fix: the taskmanager should exist for each Project to keep it project scoped. A taskmanager rewrite is needed. 

  ---

  9. Undo/redo ownership

  DataWriter.set_commit_active() exists and handles the undo logic. But who

  holds the undo stack? The old code had UndoQueue in ProjectDb. For Project2,

  this must live on the write thread. Does it belong in DataWriter itself, or as

   a separate object that Project2 holds?
   
   Fix: Actualy for now we can ignore the undo queue. undoing is juste reactivating the commits. The ui will be able to hold a local undo queue that is completely unknown to the backend.

  ---

  10. DataWriter owns its own DataReader

  DataWriter.__init__ creates self.reader = DataReader(path) and calls

  self.reader.start(). This means Project2 doesn't need to manage a separate

  reader for write-path reads (like get_max_commit_id). But it also means the

  write thread holds two SQLite connections to the same file simultaneously.

  Worth confirming this is intentional.                  

  
Fix: propose a better solution if there is.
